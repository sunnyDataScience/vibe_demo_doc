# 📊 數據分析專案檢核清單（Check List）

## 接到分析題目時的系統化流程

---

## Phase 1：問題理解 📋
```
□ 明確業務目標和成功指標
□ 了解利害關係人期望
□ 確認時間和資源限制
□ 定義分析範圍和邊界
□ 識別潛在風險和假設
```

### 關鍵問題
- **What**：要解決什麼問題？
- **Why**：為什麼這個問題重要？
- **Who**：誰會使用分析結果？
- **When**：什麼時候需要結果？
- **How**：如何衡量成功？

### Prompt 模板
```
"幫我釐清這個分析專案：
背景：[業務情境]
問題：[待解決問題]
目標：[預期成果]
限制：[時間/資源/數據]
請幫我：
1. 確認是否理解正確
2. 識別可能的盲點
3. 建議分析方向"
```

---

## Phase 2：數據評估 🔍
```
□ 盤點可用數據源
□ 評估數據品質（完整性、準確性、時效性）
□ 確認數據存取權限
□ 估算數據量和處理需求
□ 識別數據缺口
```

### 數據品質檢查維度
| 維度 | 檢查項目 | 標準 |
|:-----|:---------|:-----|
| 完整性 | 缺失值比例 | <5%可接受 |
| 準確性 | 異常值比例 | <3%可接受 |
| 一致性 | 格式統一度 | 95%以上 |
| 時效性 | 數據新鮮度 | 依業務需求 |
| 相關性 | 欄位可用性 | 核心欄位齊全 |

### Prompt 模板
```
"評估數據集[名稱]的品質：
1. 缺失值分析
2. 重複值檢測
3. 異常值識別
4. 資料類型檢查
5. 欄位相關性分析
產生品質分數和改善建議"
```

---

## Phase 3：方法設計 🎯
```
□ 選擇適當的分析類型（描述/診斷/預測/規範）
□ 設計分析框架（MECE拆解）
□ 制定假說清單
□ 選擇統計方法和模型
□ 規劃驗證方式
```

### 分析類型選擇指南
```
描述性分析 → 發生了什麼？
    適用：現狀報告、KPI監控
    方法：統計描述、視覺化

診斷性分析 → 為什麼發生？
    適用：問題根因、異常分析
    方法：相關分析、分群分析

預測性分析 → 將會發生什麼？
    適用：趨勢預測、風險評估
    方法：時間序列、機器學習

規範性分析 → 應該做什麼？
    適用：決策優化、資源配置
    方法：優化算法、模擬分析
```

### Prompt 模板
```
"為[分析目標]設計方法：
1. 推薦分析類型
2. 建議統計方法
3. 設計驗證方案
4. 預估所需時間
5. 識別技術難點"
```

---

## Phase 4：執行分析 ⚙️
```
□ 數據提取和整合
□ 數據清理和轉換
□ 探索性數據分析（EDA）
□ 執行核心分析
□ 結果驗證和敏感度分析
```

### 執行步驟細項
#### 4.1 數據準備
```
□ 載入數據（考慮記憶體優化）
□ 合併多源數據
□ 處理缺失值
□ 處理異常值
□ 特徵工程
□ 數據標準化/正規化
```

#### 4.2 探索分析
```
□ 單變數分析（分布、統計）
□ 雙變數分析（相關、交叉）
□ 多變數分析（降維、聚類）
□ 時間模式分析
□ 群組比較
```

#### 4.3 深度分析
```
□ 假說檢驗
□ 模型建立
□ 參數調優
□ 交叉驗證
□ 結果解釋
```

### Prompt 模板
```
"執行[分析類型]分析：
數據：[數據描述]
目標：[分析目標]
步驟：
1. 數據載入和預處理
2. 探索性分析
3. [特定分析方法]
4. 結果驗證
5. 視覺化呈現
請包含程式碼和解釋"
```

---

## Phase 5：洞察提煉 💡
```
□ 識別關鍵發現
□ 進行So What測試
□ 連結業務含義
□ 形成可行動建議
□ 評估影響和風險
```

### 洞察品質評估
```
✓ 可行動性：能轉化為具體行動
✓ 相關性：與業務目標相關
✓ 新穎性：提供新的視角
✓ 可驗證性：有數據支撐
✓ 影響力：對決策有實質影響
```

### What-So What-Now What 框架
| 層次 | 問題 | 輸出 |
|:-----|:-----|:-----|
| What | 數據顯示什麼？ | 事實陳述 |
| So What | 這意味著什麼？ | 業務含義 |
| Now What | 應該怎麼做？ | 行動建議 |

### Prompt 模板
```
"將分析結果轉化為洞察：
發現：[數據發現]
請提供：
1. 業務含義（So What）
2. 影響評估（量化）
3. 行動建議（具體步驟）
4. 風險提醒
5. 成功指標"
```

## Phase 6：成果交付 📈
```
□ 構建故事線
□ 準備視覺化材料
□ 撰寫執行摘要
□ 設計互動儀表板

```

```
□ 執行摘要（1頁）

   - 預期效益量化

□ 主體內容（5-10頁）
   - 背景說明
   - 方法描述
   - 分析過程
   - 詳細發現

□ 視覺化（圖表）
   - 每個論點配圖
   - 圖表自明性
   - 配色一致性

□ 附錄（技術細節）
   - 數據來源
   - 處理邏輯
   - 統計方法
   - 限制說明
```

### 溝通檢核
```
□ 受眾分析
   - 技術背景
   - 關注重點
   - 決策權限

□ 訊息設計
   - 金字塔結構
   - SCQA故事線
   - 三點原則

□ 視覺設計
   - 簡潔清晰
   - 重點突出
   - 品牌一致

□ 互動準備
   - 預期問題
   - 補充材料
   - 深入分析
```

交付形式：[報告/簡報/儀表板]

## 專案完成度評估

### 整體檢核標準
| 評估維度 | 權重 | 檢核要點 |
|:--------|:-----|:---------|
| 問題定義 | 15% | 清晰、可衡量、有價值 |
| 數據品質 | 20% | 完整、準確、相關 |
| 分析嚴謹 | 25% | 方法正確、邏輯清晰 |
| 洞察價值 | 20% | 新穎、可行動、有影響 |
| 溝通效果 | 20% | 清楚、說服力、視覺化 |

### 風險檢核
```
□ 數據風險
   - 樣本代表性
   - 倖存者偏差
   - 數據洩露

□ 分析風險
   - 過度擬合
   - 因果謬誤
   - 統計顯著性

□ 業務風險
   - 實施可行性
   - 成本效益
   - 組織接受度

□ 道德風險
   - 隱私保護
   - 公平性
   - 透明度
```

## 快速檢核卡

### 30秒檢核
```
✓ 問題清楚嗎？
✓ 數據可靠嗎？
✓ 方法合適嗎？
✓ 結論有據嗎？
✓ 建議可行嗎？
```

### 專案健康度診斷
```python
def project_health_check():
    score = 0

    # 問題定義（20分）
    if 明確的業務目標: score += 10
    if 可衡量的成功指標: score += 10

    # 數據準備（20分）
    if 數據完整性>90%: score += 10
    if 數據相關性確認: score += 10

    # 分析執行（30分）
    if 使用適當方法: score += 10
    if 完成驗證測試: score += 10
    if 考慮替代假說: score += 10

    # 結果品質（30分）
    if 洞察可行動: score += 10
    if 量化影響: score += 10
    if 風險評估: score += 10

    return score

# 評分標準
# 90-100：優秀，可以交付
# 70-89：良好，需要小調整
# 50-69：及格，需要改進
# <50：需要重新檢視
```

```
"我正在進行[分析類型]專案，
目前在[階段名稱]階段，

請幫我：
1. 用上述檢核清單評估完整性
2. 識別可能遺漏的關鍵步驟

```

---

## 使用說明
1. 專案開始時通讀一遍，了解全貌
2. 每個階段開始前檢查相應章節
3. 完成後打勾，追蹤進度
4. 交付前做最終檢核
5. 專案結束後回顧改進
