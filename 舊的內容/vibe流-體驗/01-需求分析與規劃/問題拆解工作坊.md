# 🔍 問題拆解工作坊：學會結構化思考

## 🎯 為什麼要學拆解問題？
想像你要搬家，如果只想著「我要搬家」，你會不知道從何開始。但如果拆解成：
1. 找新房子
2. 打包物品
3. 聯絡搬家公司
4. 辦理地址變更

每個步驟都變得可以執行了！軟體開發也是一樣。

## 🏗️ 拆解的三個層次

### 第一層：問「為什麼」(Why)
**目的**：確認我們真正要解決的問題

**範例**：
- 原始需求：「我想要一個新聞摘要工具」
- 問為什麼：「為什麼需要摘要工具？」
- 真正問題：「因為每天資訊太多，沒時間全部看完」

**練習**：
```
我的需求是：_________________
我為什麼需要這個：_________________
真正的問題是：_________________
```

### 第二層：問「什麼」(What)
**目的**：定義具體要做什麼

**思考框架**：
1. **輸入**：用戶會提供什麼？
2. **處理**：系統要做什麼？
3. **輸出**：用戶會得到什麼？

**範例**：
- 輸入：網址清單 + 個人喜好
- 處理：抓取新聞 + AI摘要 + 定時執行
- 輸出：摘要文字 + 語音播報 + 原文連結

### 第三層：問「怎麼做」(How)
**目的**：確定實現的方法

**拆解原則**：
- **由大到小**：大功能 → 小功能 → 具體任務
- **由外到內**：用戶體驗 → 系統架構 → 技術細節
- **按時間順序**：第一步 → 第二步 → 第三步

## 🛠️ 實戰演練：新聞摘要助手拆解

### Step 1: 用戶旅程拆解
```
1. 用戶打開Chrome插件
2. 輸入新聞網址清單
3. 設定摘要偏好（比如：商業角度分析）
4. 選擇執行時間（現在 or 排程）
5. 獲得摘要結果
6. 選擇聆聽語音版本
7. 點擊查看原文
```

### Step 2: 技術功能拆解
```
前端部分：
├── 插件介面設計
├── 用戶輸入處理
├── 結果展示頁面
└── 設定管理

後端部分：
├── 網頁內容抓取
├── AI摘要生成
├── 排程任務管理
├── 語音合成
└── 數據存儲
```

### Step 3: 開發任務拆解
```
第一週：
├── 建立專案架構
├── 設計資料庫結構
└── 完成基礎插件框架

第二週：
├── 實現網頁抓取功能
├── 整合AI摘要API
└── 建立簡單介面

第三週：
├── 新增排程功能
├── 實現語音合成
└── 優化用戶體驗
```

## 🎯 拆解檢查清單
完成拆解後，檢查是否滿足：

- [ ] **具體可行**：每個任務都知道怎麼做
- [ ] **時間合理**：每個任務不超過一天工作量
- [ ] **相互獨立**：任務之間依賴關係清楚
- [ ] **可以測試**：每個功能都能驗證是否完成
- [ ] **優先順序**：知道哪些重要，哪些可以延後

## 💡 拆解技巧

### 1. 5W1H法
- Who（誰）：誰會使用？
- What（什麼）：要做什麼功能？
- When（何時）：什麼時候需要？
- Where（哪裡）：在什麼環境使用？
- Why（為什麼）：為什麼需要？
- How（怎麼做）：如何實現？

### 2. 金字塔原理
```
大目標
├── 子目標1
│   ├── 具體任務1-1
│   └── 具體任務1-2
├── 子目標2
│   ├── 具體任務2-1
│   └── 具體任務2-2
└── 子目標3
    ├── 具體任務3-1
    └── 具體任務3-2
```

### 3. MVP思維
**M**inimum（最小）**V**iable（可行）**P**roduct（產品）
- 先做最核心的功能
- 後續再加強和擴展
- 快速驗證想法是否可行
